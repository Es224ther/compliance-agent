# Compliance Agent：面向全球化 AI 产品的合规审计 Agent

## 产品需求文档 (PRD) 

---

## 1. 背景

在 AI 产品快速迭代过程中，跨国数据流转与 AIGC 合规的要求日益严苛。产品经理、安全治理、法务和研发团队经常需要对新功能进行合规评估，例如：

- 用户数据是否可以用于模型训练
- 是否涉及跨境传输
- 是否存在敏感个人信息处理
- 是否需要额外告知、同意或风险评估

当前这类评估往往依赖人工查阅法规、反复沟通，成本高、效率低，容易导致需求推进缓慢或遗漏高风险点。

本项目希望构建一个**面向全球化 AI 产品合规审计助手**，通过自然语言输入业务场景，系统自动进行结构化解析、法规证据检索（RAG）并输出包含可解释性（XAI）的初步合规整改建议，加速 AI 产品的合规上线流程。

---

## 2. 目标受众和用户画像

| 角色 | 核心诉求 | 使用频率 | 技术水平 |
|------|---------|---------|---------|
| AI 产品经理 | 在需求规划期快速验证功能的合规可行性，避免需求上线后被法务"打回" | 每周 2-3 次 | 中等，熟悉业务但不精通法规条文 |
| 安全治理专员 | 对即将上线的 AI 功能进行风险初筛，生成可交付法务的预评估报告 | 每周 1-2 次 | 较高，了解 GDPR/PIPL 框架但难以追踪所有细则更新 |
| AI 研发工程师 | 评估数据采集方案与大模型 API 调用的底层合规要求（如脱敏、日志留存） | 按需 | 高技术水平，但对法律条款理解有限 |

## 3. 产品边界与法规范围

### 3.1 In-Scope

本版本仅覆盖**欧洲（EU）与中国（CN）**的核心数据与 AI 治理框架：

- **EU**：GDPR（隐私与数据跨境）、EU AI Act（高风险 AI 与透明度）
- **CN**：PIPL（个保法）、DSL（数据安全法）、CSL（网络安全法）及 2025 年 9 月全面生效的《人工智能生成合成内容标识办法》

**商业逻辑**：中欧代表了全球最严苛的隐私监管与最复杂的 AI 安全体系。能够模拟出海业务最典型的"合规夹击"，最大化验证 Agent 的深度推理能力。

### 3.2 Out of Scope

- 北美（US）及其他亚太地区（APAC）法规（规划至 Phase 2）
- 输出具备法律效力的正式意见（仅作为业务前置风控参考）
- 全自动的合规阻断与审批流

### 4.2 竞争定位与差异化

**市场空白**：现有产品要么是"重量级平台"（Securiti、OneTrust），部署成本高、流程复杂；要么是"单法域工具"（GDPRLocal），法域覆盖不足。**没有一款产品专门解决"AI 产品经理在功能设计阶段，用自然语言描述一个业务场景，30 秒内获得中欧双法域合规预检报告"这个痛点。**

**我们的差异化定位**：

| 维度 | 现有竞品 | Compliance Agent |
|------|---------|-----------------|

| 技术架构 | SaaS 平台 | 轻量 Agent（RAG + Guardrails） |

## 5. 用户场景

### 场景 A：跨境数据训练

产品团队计划将欧洲用户上传的视频素材用于训练生成式剪辑模型，希望评估相关数据处理方案的合规性。重点关注两方面：一是该类数据在跨境流转过程中是否满足 GDPR 关于数据出境的合法性要求；二是相关数据在中国境内使用时，是否符合《数据安全法》（DSL）对数据处理活动的合规要求。

### 场景 B：第三方模型 API 调用

开发团队希望接入第三方大模型 API，对用户提交的文本与图片内容进行处理。在方案落地前，需要评估该过程是否构成向第三方提供或共享数据，以及由此可能引发的合规管理风险。

### 场景 C：AIGC 内容标识

业务团队计划向欧洲 B 端客户提供 AI 自动生成广告视频的 API。系统设计需要同时满足多法域合规要求：一方面，应对生成内容提供明确提示，以履行 EU AI Act 对深度伪造（deepfake）透明度义务；另一方面，还需在底层生成文件中注入符合中国 AIGC 合规要求的隐式元数据标签（Metadata）。

## 6. 用户交互流程

### 6.1 核心用户旅程（以 场景 A 为例）

**阶段 1：场景输入**

用户以自然语言描述业务场景。系统提供输入引导框（Placeholder），示例如下：

> "请描述你的 AI 功能场景，包括：涉及哪些用户数据、数据在哪些地区流转、是否使用第三方模型、AI 生成内容是否面向终端用户。"

用户输入示例：
> "我们计划把欧洲用户上传的短视频素材传回国内服务器，用于训练一个 AI 视频剪辑模型。训练完成后模型部署在国内，服务全球用户。"

**阶段 2：系统处理（对用户透明但不需操作）**

- **Step 2a — PII 本地脱敏**：系统在本地对输入文本进行 PII 扫描和掩码处理（如将具体人名、邮箱、IP 替换为占位符），然后才将脱敏后的文本发送给云端 LLM。
- **Step 2b — 场景结构化解析**：LLM 从自然语言中提取结构化字段。
- **Step 2c — 信息完整性校验**：系统检查提取字段的完整性。

**阶段 3：智能追问（条件触发）**

当关键字段缺失或模糊时，Agent 拒绝直接给出判断，转而以结构化问题追问用户：

> "为了给出准确的合规评估，我需要确认以下信息：
> 1. 视频素材中是否包含可识别的人脸或生物特征信息？ [是 / 否 / 不确定]
> 2. 数据传输是否经过欧盟境内的中间服务器？ [是 / 否]
> 3. 贵司是否已取得用户对数据跨境传输的明确同意？ [已取得 / 未取得 / 部分取得]"

**交互设计原则**：追问最多 2 轮，每轮不超过 3 个问题。若 2 轮后信息仍不完整，系统在报告中标记"信息不足项"并标注为"需人工法务补充评估"。

**阶段 4：法规检索与分析**

系统基于结构化字段在本地法规知识库中进行 RAG 检索，匹配相关法规条款，并由 LLM 进行风险推理与建议生成。

**阶段 5：报告输出与交付**

系统生成 5 段式审计报告（详见 FR6），支持以下交互：

- **在线预览**：Markdown 渲染的结构化报告
- **导出格式**：Markdown / JSON（Phase 1）；PDF（Phase 2）
- **一键分享**：生成可分享链接，方便转发给法务同事进行二次复核
- **反馈机制**：用户可对报告中的每条建议标记"有帮助 / 无帮助 / 需修改"，用于后续模型优化

## 7. 功能需求

### FR1. 场景摄入与本地数据脱敏

用户输入自然语言业务描述。

**安全要求**：系统在将上下文传给云端大模型前，必须通过本地轻量级 NLP 模型对文本中的个人敏感信息（PII，如姓名、邮箱、IP）进行掩码屏蔽（Masking），满足最小化处理原则。

### FR2. 场景结构化解析

系统需将自然语言提取为确定性的强类型字段：

- `region`：EU, CN, Global
- `data_types`：Personal, Behavioral, Biometric, etc.
- `cross_border`：Boolean
- `third_party_model`：Boolean
- `aigc_output`：Boolean（是否涉及 AI 生成内容面向终端用户）
- `data_volume_level`：Small / Medium / Large（影响是否触发数据安全评估）

### FR3. 证据驱动的检索

基于解析字段，在本地知识库中精准匹配并提取相关的法规原文切片（Chunks），提供法规名称、具体条款及相关度评分。

### FR4. 智能护栏与拒答逻辑

- **信息不足**：当关键字段（如数据流向）缺失时，Agent 必须触发追问，拒绝在信息模糊时给出判断。
- **置信度低**：当检索不到充分法规证据时，标记为"需人工法务复核（Escalation）"。
- **越界拒答**：当用户要求系统输出具有法律效力的正式意见时，明确告知系统定位为"前置风控参考"。

### FR5. 风险定级与整改建议生成

基于可解释性 AI（XAI）逻辑输出：

- **Risk Level**：Low / Medium / High / Critical
- **Actionable Advice**：按角色提供差异化建议
  - 产品经理：需要补充的授权弹窗、隐私政策更新
  - 研发工程师：需要增加的本地去标识化管道、Metadata 注入
  - 安全治理：需要启动的 DPIA 或安全评估流程

### FR6. 报告导出

系统生成并展示固定结构的 5 段式审计报告（Markdown/JSON）：

1. **场景摘要（Summary）**：用结构化语言复述用户场景
2. **风险等级（Risk Level）**：Low / Medium / High / Critical + 一句话风险概述
3. **证据引用（Evidence Citations）**：每条判断引用的法规名称、条款编号和原文摘要
4. **不确定项/待定项（Uncertainties）**：信息不足或置信度低的判断点
5. **落地整改建议（Remediation Actions）**：按角色分组的可执行建议清单

## 8. 技术方案设计

### 8.1 整体流水线

```
用户输入 → [本地 PII 脱敏] → [LLM 场景解析] → [RAG 法规检索] → [护栏校验] → [风险分析 + 报告生成] → 输出
```

### 8.2 FR1 技术方案：PII 脱敏模块

**选型：Microsoft Presidio + 中文 NER 扩展**

Microsoft Presidio 是微软开源的 PII 检测和匿名化框架，采用正则表达式、命名实体识别（NER）和上下文感知检测三层机制。Presidio 默认使用 spaCy 的 `en_core_web_lg` 模型作为英文 NER 引擎，支持通过自定义配置扩展多语言能力。

**中文 PII 识别方案**：

由于本产品涉及中欧双语场景，需要对 Presidio 进行中文扩展：

- **NLP 引擎层**：加载 spaCy 的中文 NLP Pipeline（如 `zh_core_web_trf`），在 Presidio 的 `NlpEngineProvider` 配置文件中注册中文语言支持。
- **识别器层**：为中文场景新建 `PatternRecognizer`，覆盖中国大陆手机号（`1[3-9]\d{9}`）、身份证号（18 位校验）、中文姓名等 PII 类型，并配置中文上下文词（如"姓名"、"手机"、"身份证"）以提升置信度。
- **本地部署**：PII 脱敏模块必须在本地运行，脱敏后的文本才可发送至云端 LLM，确保原始 PII 不出域。

**脱敏策略**：采用占位符替换（如 `[PERSON_1]`、`[EMAIL_1]`），而非不可逆哈希，以便后续报告中还原为可读的泛化描述。

### 8.3 FR2 技术方案：场景结构化解析

**选型：LLM Function Calling / Structured Output**

利用大模型的 Function Calling 或 JSON Schema 约束输出能力，将自然语言映射为强类型字段。

**实现方式**：

- 定义 JSON Schema，包含所有需要提取的字段（region、data_types、cross_border 等），每个字段带有 `enum` 约束和 `description`。
- 通过系统提示词（System Prompt）注入解析规则和少量示例（Few-shot），引导 LLM 输出符合 Schema 的 JSON。
- LLM 输出后，使用 JSON Schema Validator 进行格式校验；不符合的字段回退为 `null`，触发 FR4 追问逻辑。

**解析准确率保障策略**：

- 内置 20+ 标注好的测试用例（覆盖三大 Scenario 的变体），持续回归测试 Parse Success Rate。
- 对于 LLM 无法确定的字段（如 `data_types` 中是否包含 Biometric），允许返回 `"uncertain"`，交由追问逻辑处理，而非强制猜测。

### 8.4 FR3 技术方案：法规知识库构建与 RAG 检索

**知识库构建**：

法规文档的切片策略（Chunking Strategy）直接决定 RAG 检索质量。法规文本具有独特的层级结构（篇 → 章 → 条 → 款 → 项），且条款之间存在大量交叉引用，不能简单按固定字数切分。

**切片策略：层级感知切片（Hierarchy-Aware Chunking）**

- **切片单元**：以"条"（Article）为基本切片单元，每条法规条款形成一个独立的 Chunk。
- **元数据注入**：每个 Chunk 附带结构化元数据标签：法规名称（如 `GDPR`）、条款编号（如 `Article 46`）、所属章节（如 `Chapter V: Transfers of personal data to third countries`）、法域（`EU` / `CN`）、关键词标签（如 `cross_border_transfer`、`adequacy_decision`）。
- **摘要增强（Summary-Augmented Chunking）**：参考学术界的 SAC 方法，为每个 Chunk 附加一段法规级别的摘要上下文（约 150 字），帮助检索模型理解该条款在整体法规中的定位，减少"文档级检索偏差"（Document Retrieval Mismatch）。
- **交叉引用处理**：对法规中引用其他条款的内容（如 GDPR Art. 49 引用 Art. 46），在 Chunk 的元数据中建立引用关系图，检索时可自动关联被引用条款。

**法规范围（Phase 1）**：

| 法规 | 条款数量（估计） | 语言 |
|------|---------------|------|
| GDPR | ~99 条 | 英文 |
| EU AI Act | ~113 条 + 附件 | 英文 |
| PIPL（个保法） | ~74 条 | 中文 |
| DSL（数据安全法） | ~55 条 | 中文 |
| CSL（网络安全法） | ~79 条 | 中文 |
| 《AIGC 标识办法》 | ~25 条 | 中文 |

总计约 450+ 条款，加上摘要和元数据，估计知识库规模约 2000-3000 个 Chunks。

**检索方案：混合检索（Hybrid Retrieval）**

- **语义检索**：使用多语言嵌入模型（如 `multilingual-e5-large` 或 `bge-m3`），对用户场景的结构化字段进行语义向量匹配，召回 Top-10 候选 Chunks。
- **关键词检索**：使用 BM25 对法规条款编号、关键术语（如"Standard Contractual Clauses"、"数据出境安全评估"）进行精确匹配，弥补语义检索对专业术语的不敏感。
- **重排序（Re-ranking）**：使用 Cross-Encoder 重排序模型（如 `bge-reranker-v2-m3`），对混合召回的候选 Chunks 进行精排，选取 Top-3 至 Top-5 作为最终证据输入 LLM。

**双语处理策略**：

- 用户输入为中文时，检索同时覆盖中文法规（PIPL/DSL/CSL）和英文法规（GDPR/EU AI Act），因为中欧双法域场景需要同时引用。
- 嵌入模型选用跨语言模型，确保中文 Query 可以匹配英文法规条款。
- 最终报告中，法规引用保留原文语言，但建议文字统一使用用户输入语言。

### 8.5 FR4 技术方案：护栏与拒答

- **字段完整性规则引擎**：硬编码的必填字段校验（region、data_types、cross_border 为必填），缺失时自动生成追问 Prompt。
- **检索置信度阈值**：当 Re-ranker 返回的最高分 Chunk 分数低于设定阈值（如 0.6），或者 Top-3 Chunks 来自不同法规且无交集，标记该判断为低置信度，触发 Escalation 标签。
- **输出安全过滤**：对 LLM 生成的建议文本进行后处理扫描，拦截任何可能被解读为"法律意见"的表述（如"贵司必须"、"违法"等绝对化用语），替换为更审慎的措辞。

### 8.6 LLM 选型考量

| 考量维度 | 选型建议 |
|---------|---------|
| 主模型 | Claude Sonnet 或 GPT-4o-mini（平衡能力与成本） |
| 上下文窗口 | ≥ 32K Tokens（法规 Chunks + 系统提示 + 场景描述） |
| 结构化输出 | 要求支持 Function Calling 或 JSON Mode |
| 成本控制 | 单次查询控制在 $0.01-0.05 范围内 |

## 9. 成功指标

### 9.1 产品体验指标

| 指标 | 目标值 | 测量方法 |
|------|-------|---------|
| 端到端延迟 | 场景输入后 ≤ 30 秒返回完整报告 | 从用户点击"开始评估"到报告渲染完成的时间 |
| 解析成功率（Parse Success Rate） | ≥ 90% 的测试用例成功提取核心结构化字段 | 基于 50+ 标注测试集的自动化回归测试 |
| 追问触发准确率 | 在信息残缺测试集中 ≥ 80% | 故意输入缺失字段的场景，验证追问是否正确触发 |

### 9.2 合规与安全质量指标

| 指标 | 目标值 | 测量方法 |
|------|-------|---------|
| 证据覆盖率（Evidence Coverage） | ≥ 80% 的 High/Critical 风险判定包含有效知识库法规引用 | 人工审核报告中的证据引用是否指向真实法规条款 |
| 幻觉率（Hallucination Rate） | ≤ 10% 的法规引用为无效或不存在的条款 | 人工核查 + 自动化条款存在性校验 |
| PII 脱敏覆盖率 | ≥ 95% 的输入文本中 PII 被正确识别和掩码 | 基于含 PII 标注的测试集回归测试 |

### 9.3 用户价值指标

| 指标 | 目标值 | 测量方法 |
|------|-------|---------|
| 合规评估周期缩短比 | 相比纯人工评估缩短 ≥ 60% | 对比基线：当前人工合规初评平均耗时（预计 2-5 天） |
| 报告采纳率 | ≥ 70% 的报告被用户标记为"有帮助" | 用户反馈机制收集 |
| 报告分享率 | ≥ 30% 的报告被用户通过分享链接转发 | 分享链接点击追踪 |

### 9.4 与法务专家一致性评估

在 MVP 阶段，选取 20 个典型场景同时由 Agent 和法务专家进行评估，计算：

- **风险等级一致率**：Agent 的 Risk Level 与法务专家判断一致的比例，目标 ≥ 70%
- **关键风险点召回率**：法务专家识别的高风险点中，Agent 也识别到的比例，目标 ≥ 80%

## 10. 风险与局限性

### 10.1 法规与合规风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 法规更新时效性 | 中国 AI 法规变动频繁（如《AIGC 标识办法》2025 年 9 月生效，CAC 持续发布细则），知识库可能滞后 | 建立法规更新监控机制：Phase 1 采用人工季度审查 + 更新；Phase 2 引入 RSS/API 半自动监控主要立法机关动态 |
| 法规适用的上下文依赖 | 同一场景在不同业务上下文中可能适用不同法规条款，系统难以完全捕获 | 报告中明确标注"本评估基于用户描述的场景，实际合规义务可能因具体业务上下文而异"；高风险场景强制标记"需人工法务复核" |
| 跨法域冲突 | GDPR 与 PIPL 在某些场景下的合规要求可能存在张力（如 GDPR 要求 SCCs 而 PIPL 要求安全评估），系统需正确呈现而非简单合并 | 报告中分法域独立呈现合规要求，并增加"跨法域注意事项"段落，明确提示潜在冲突 |

### 10.2 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 幻觉（Hallucination） | 生成不存在的法规条款或错误的合规建议 | RAG 证据约束 + 输出后处理校验（检查引用条款是否存在于知识库）+ Escalation 机制 |
| 双语检索准确率 | 中文场景匹配英文法规条款时可能出现语义偏差 | 使用专门的跨语言嵌入模型 + 为法规 Chunks 添加双语关键词标签 + 混合检索弥补语义检索不足 |
| 中文 PII 识别不全 | Presidio 默认配置对中文 PII 支持有限 | 自建中文 PatternRecognizer + 在标注数据集上持续测试和调优 |
| 解析失败的长尾场景 | 用户输入极度模糊或包含专业术语时解析失败 | 设计优雅降级：解析失败时提供模板化追问，而非返回错误 |

### 10.3 产品与运营风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 用户过度信赖 | 用户可能将 Agent 输出视为法律意见，跳过人工法务审核 | 每份报告顶部和底部固定显示免责声明；High/Critical 风险报告强制显示"本报告不替代专业法律意见" |
| Agent 自身合规性 | Agent 处理用户输入的业务描述可能包含商业机密 | 本地 PII 脱敏 + 云端 LLM 调用使用零数据留存（Zero Data Retention）策略 + 系统自身通过数据处理影响评估（DPIA） |
| 知识库维护成本 | 法规更新需要人工介入重新切片和标注 | Phase 1 接受季度更新周期；Phase 2 探索半自动化法规变更检测和差分更新 |

### 10.4 明确的产品边界声明

- 本项目**不替代**法务审批和正式 DPIA
- 本项目**不输出**具有法律效力的正式意见
- 对复杂跨境和高风险 AI 场景，**仍需人工复核**
- 检索与生成仍存在误判风险，所有输出仅作**初步参考**
